# Codebase Indexer Skill

A skill for reading a codebase and producing a structured Markdown index file — a single source of truth that tells an AI *what exists*, *what it does*, and *how pieces connect* — so future AI sessions can answer questions, make changes, or debug without re-reading the whole repo.

---

## Core Principle

The output is not documentation for humans. It is a **navigation index for AI**. Every section should answer the question: *"If an AI needs to do X, where should it look and what should it know first?"*

This means:
- Prefer **precise file paths** over vague descriptions
- Prefer **concrete examples** (function signatures, data shapes) over prose explanations
- Every entry in the index should be **actionable** — it tells the AI what to touch, not just what exists

---

## Workflow

### Step 1: Explore Project Structure

Before writing anything, map the full directory tree. Run or simulate:

```bash
find . -type f | grep -v node_modules | grep -v .git | grep -v __pycache__ | grep -v .env | sort
```

Or if files are provided by the user, list them all first.

Then identify:
- **Entry points** — where does the program start? (main.py, index.js, app.ts, server.go, etc.)
- **Configuration files** — .env.example, config/, docker-compose.yml, package.json, pyproject.toml
- **Core directories** — what does each top-level folder contain?
- **Test locations** — where are tests? what framework?
- **Build/deploy artifacts** — Dockerfile, CI configs, Makefile

---

### Step 2: Analyze Each Layer

For each major module or directory, extract:

1. **Purpose** — one sentence, what is this module responsible for?
2. **Key files** — the 2–5 files that matter most (not every file)
3. **Public interface** — functions/classes/routes that other modules call
4. **Dependencies** — what does this module import from? what does it export to?
5. **Data shapes** — key types, schemas, or models defined here

Skip boilerplate files (auto-generated, lock files, build outputs). Focus on files a developer would actually edit.

---

### Step 3: Identify Cross-Cutting Concerns

Look for patterns that span multiple modules:

- **Auth / authorization** — how is identity established and checked?
- **Error handling** — is there a central error format or handler?
- **Logging** — what library, what format, where does it output?
- **Data flow** — how does data move from input (API/CLI/event) to output (DB/response/file)?
- **State management** — where is shared state kept? (DB, cache, in-memory store)
- **External integrations** — third-party APIs, queues, services called

---

### Step 4: Write the Index File

Produce a single `PROJECT_INDEX.md` using the template below. Fill every section based on what was found. If a section doesn't apply, write `N/A — [brief reason]`.

## Quality Rules for the Output

Before finishing, verify the index passes these checks:

- [ ] Every directory in the project is explained (even if just "auto-generated, ignore")
- [ ] Every module has at least one key file listed with a role description
- [ ] The data flow section traces at least one real end-to-end path
- [ ] The AI Navigation Guide covers at least 6 common tasks
- [ ] No section says just "see code" — if something is important, it's described here
- [ ] File paths are exact (copy from the actual file listing, don't guess)
- [ ] No weasel words: "handles things", "manages stuff", "does X-related logic" — be specific

---

## Closing Behavior

After producing `PROJECT_INDEX.md`:

1. State which parts had low confidence (files you couldn't read fully, unclear patterns)
2. List any assumptions made (e.g. "Assumed standard MVC structure based on folder names")
3. Offer to go deeper on any specific module if the user needs more detail
4. Remind the user: the index should be regenerated when major structural changes happen
